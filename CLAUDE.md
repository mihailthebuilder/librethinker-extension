# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

LibreThinker is a LibreOffice Writer sidebar extension (`.oxt`) written in Python via the UNO API. It adds a panel where the user types a prompt and the extension rewrites either the selected text or the entire document using an LLM. Three LLM paths are supported:

- **Free model** — proxied through `https://api.librethinker.com` (no model/key provided).
- **BYOK** — same backend, but with a user-supplied `model` ID (LiteLLM format, e.g. `gpt-5.2`, `mistral/mistral-medium`) and API key forwarded via the `X-Third-Party-Key` header.
- **Ollama / self-hosted** — model IDs prefixed `sh/ollama/` bypass the backend and hit `http://localhost:11434/api/chat` (or a custom `Model URL`) directly.

## Build / package

The build depends on an external tool, **unodit**, expected as a sibling checkout at `../unodit/` (i.e. `../unodit/unodit.py`). The `Makefile` also hardcodes the absolute path `C:\Users\mihai\projects\librethinker-extension\extension` — update these for another machine.

- `make unodit-zip` — package into `dist/LibreThinker.oxt` (`sidebar_oxt`). **This is the only target used in normal development.**
- `make unodit-code` — regenerate `Panel1_UI.py` from the XDL dialog (`sidebar_convert`). This was used **once** for the initial scaffolding of the panel code and is **not** intended to be run again — `Panel1_UI.py` is now maintained by hand (see Architecture). Do not run this; it would overwrite the hand-edited UI file.
- `make unodit-files` — regenerate the extension metadata/config files (`sidebar_files`). Also a one-time scaffolding step, not part of the normal workflow.

Install the produced `.oxt` via LibreOffice *Tools > Extension Manager > Add*. There is no test suite, linter, or CI in this repo.

Target runtime is Python 3.11 (`.python-version`) — but actual execution uses LibreOffice's bundled Python interpreter, not a local venv.

## Architecture

Entry point is `extension/src/LibreThinker.py`, registered as a UNO component in `META-INF/manifest.xml`. It implements the sidebar factory (`ElementFactory` / `XUIPanel`) and the `More Options` dispatch handler. When the sidebar panel becomes visible, `showPanels()` instantiates `Panel1`.

Code under `extension/src/pythonpath/` (on the UNO `pythonpath`):

- `ui/Panel1_UI.py` — defines all dialog controls (Prompt, Submit, radio buttons, settings fields) and dispatches action commands to methods on `Panel1`. It was originally generated by unodit, but is now **edited by hand** — unodit is no longer re-run against it (see Build / package). Change the UI by editing this file directly.
- `ui_logic/Panel1.py` — the real controller. Reads document text, wires the Submit/SaveSettings buttons, calls the API clients on a background thread, and writes the response back into the current selection.
- `ui_logic/api.py` — `LtClient` (LibreThinker backend, async job: POST to `jobs/free` or `jobs/byok`, then poll `jobs/fetch` with backoff until `successful`/`failed`/timeout) and `OllamaClient` (single synchronous chat call).
- `ui_logic/settings.py` — persists `modelId`/`apiKey`/`modelUrl` as JSON to `LibreThinkerConfig.json` in LibreOffice's UserConfig dir.
- `ui_logic/utils.py` — version comparison and `sh/ollama/` prefix helpers.

### Things to know when editing

- `Panel1_UI.py` is hand-maintained; keep it limited to UI control definitions and action-command wiring — any real logic belongs in `Panel1.py`, not the UI file.
- Submit runs on a `threading.Thread` (`submit_background`) so the UI doesn't freeze; the Submit button is disabled for the duration and re-enabled in a `finally`. Note that UNO control mutation from a background thread is done directly here.
- "Entire document" mode exports the doc to a temp `.txt` (UTF-8) and reads it back, rather than walking the UNO text model. "Selected text" mode reads/writes `CurrentController.getSelection()`.
- The backend enforces a `CHARACTER_LIMIT` of 110,000 chars (prompt + text), checked client-side before submitting.

### Version bumps

The extension version is duplicated in **three** places and they must stay in sync:
- `extension/description.xml` (`<version value=.../>`)
- `extension/config.ini` (`[extension] version =`)
- `extension/src/pythonpath/ui_logic/Panel1.py` (`extensionVersion = "..."` in `server_response`)

The backend returns the latest published version (`Client-Version-Latest` header); the UI shows an "update available" hint when the local version is older.
